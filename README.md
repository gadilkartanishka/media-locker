# Media Locker

A full-stack application that allows users to upload images and monetize them by setting an unlock price. Other users can only see a blurred preview until they spend coins from their wallet to unlock the original content.

## Tech Stack

- **Frontend:** React (Vite)
- **Backend:** Node.js, Express
- **Database:** PostgreSQL
- **Storage:** s3rver (local S3-compatible object storage)
- **Auth:** JWT

## Setup Instructions

### Prerequisites

- Node.js v18+
- PostgreSQL

### 1. Clone the repo

```
git clone https://github.com/gadilkartanishka/media-locker.git
cd media-locker
```

### 2. Database Setup

```
psql
CREATE DATABASE media_locker;
\c media_locker
\i server/src/db/schema.sql
\q
```

### 3. Backend Setup

```
cd server
npm install
```

Create a `.env` file inside the `server` folder:

```
PORT=3000
DB_USER=your_postgres_username
DB_HOST=localhost
DB_NAME=media_locker
DB_PASSWORD=your_postgres_password
DB_PORT=5432
JWT_SECRET=yoursecretkey123
```

Start the server:

```
node src/index.js
```

The backend runs on http://localhost:3000. S3rver starts automatically on port 4568.

### 4. Frontend Setup

```
cd client
npm install
npm run dev
```

Open http://localhost:5173

## Security Decisions

### Private Media Storage

Original images are never publicly accessible. They are stored in a private S3 bucket. The bucket is not publicly exposed — files can only be accessed via presigned URLs generated by the server.

### Temporary Access URLs (Presigned URLs)

When a user requests media they have access to, the server generates a presigned URL that expires in 5 minutes. This means:

- Direct S3 URLs cannot be shared — they expire quickly
- The frontend never has permanent access to original files
- Every request generates a fresh short-lived URL

### Access Control

Before generating any presigned URL for an original image, the server verifies:

- Is the user the uploader? or
- Has the user paid to unlock this media?

If neither condition is true, original_url is returned as null.

### Preventing Direct Access to Original Files

Even if someone discovers the S3 key of an original image, they cannot access it without a valid presigned signature. The signature is generated using a secret key only the server knows.

### Ownership Validation

The server checks uploader_id on every media request. Owners can always access their own content without paying. Owners cannot unlock their own media (no self-purchase).

### Duplicate Purchase Prevention

The unlocks table has a UNIQUE(user_id, media_id) constraint at the database level. This prevents duplicate purchases even under concurrent requests.

### Atomic Unlock with Database Transactions

The unlock flow uses a PostgreSQL transaction — coin deduction, transaction recording, and unlock record creation all happen atomically using BEGIN / COMMIT / ROLLBACK. If any step fails, everything is rolled back. This prevents scenarios like coins being deducted without access being granted.

### Password Security

Passwords are hashed using bcrypt with 10 salt rounds. Plain text passwords are never stored.

### JWT Authentication

All protected routes require a valid JWT token in the Authorization: Bearer <token> header. Tokens expire after 7 days.

## Client-Side Image Processing

Before upload, images are processed in the browser:

- **File validation** — file type (images only) and size (max 10MB) are checked before the request is sent
- **Compression** — images are compressed to max 1MB and 1920px using browser-image-compression
- **Preview** — user sees a preview of the image with compression info before uploading

## Frontend Screens

The lightweight React application contains the following screens:

- **Login / Register** — Handles user onboarding and login, persisting JWT tokens and local session states.
- **Media Feed** — Displays all available uploads, prices in coins, and visual indicators of status (Locked / Unlocked / User's own uploads).
- **Upload Media** — Prompts the user to set a title and coin unlock price, validates the image, and compresses it client-side before sending.
- **Media Details** — A detail view where users can unlock the content with one click or view the full-quality original image if already unlocked or owned.
- **Wallet Details** — Displays the user's current coin balance and a chronological log of all transactions (debiting unlocks and crediting events).

## API Documentation

All protected endpoints require:

```
Authorization: Bearer <token>
```

---

### Auth

#### POST /auth/register

Register a new user. Every user starts with 100 coins.

Request body:

```
{
  "name": "John Doe",
  "email": "user@test.com",
  "password": "123456"
}
```

Response 201:

```
{
  "token": "eyJ...",
  "user": {
    "id": 1,
    "name": "John Doe",
    "email": "user@test.com",
    "coins": 100
  }
}
```

---

#### POST /auth/login

Login with existing credentials.

Request body:

```
{
  "email": "user@test.com",
  "password": "123456"
}
```

Response 200:

```
{
  "token": "eyJ...",
  "user": {
    "id": 1,
    "name": "John Doe",
    "email": "user@test.com",
    "coins": 100
  }
}
```

---

### Media

#### GET /media

Get all media with locked/unlocked status for the current user. Returns presigned blurred URLs for previews.

Response 200:

```
{
  "media": [
    {
      "id": 1,
      "title": "test image",
      "unlock_price": 10,
      "blurred_url": "http://localhost:4568/...",
      "original_url": "http://localhost:4568/...",
      "is_unlocked": false,
      "is_owner": true,
      "created_at": "2026-06-05T12:00:00.000Z"
    }
  ]
}
```

Note: original_url is null for users who have not unlocked the media and are not the owner.

---

#### GET /media/:id

Get a single media item with presigned URLs.

Response 200:

```
{
  "id": 1,
  "title": "test image",
  "unlock_price": 10,
  "is_unlocked": true,
  "is_owner": false,
  "blurred_url": "http://localhost:4568/...",
  "original_url": "http://localhost:4568/..."
}
```

---

#### POST /media/upload

Upload a new image. Requires multipart/form-data.

Fields:

- title — string
- unlock_price — number (coins)
- image — file (image only, max 10MB, compressed client-side before sending)

Response 201:

```
{
  "media": {
    "id": 1,
    "title": "test image",
    "unlock_price": 10,
    "original_key": "originals/...",
    "blurred_key": "blurred/...",
    "created_at": "2026-06-05T12:00:00.000Z"
  }
}
```

---

#### POST /media/:id/unlock

Unlock a media item by spending coins.

Response 200:

```
{
  "message": "Media unlocked successfully"
}
```

Error responses:

- 400 — Already unlocked
- 400 — Insufficient coins
- 400 — You cannot unlock your own media
- 404 — Media not found
